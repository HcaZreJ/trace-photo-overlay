# 轨迹照片九宫格 · trace-photo-overlay

把**任意运动轨迹**（骑行 / 徒步 / 越野跑 / 跑步 / 滑雪…）叠加到九宫格照片上，一键导出朋友圈九图。

**纯前端 · 零后端 · 零上传 · 完全离线。** 单个 HTML 文件即是全部，照片与轨迹永不离开你的设备。

**在线使用**：https://hcazrej.github.io/trace-photo-overlay/ （部署后约几分钟生效）

> GPX/KML/GeoJSON 是通用的轨迹格式——任何记录了 GPS 轨迹的运动都能用，不限于骑行。

## 怎么用

**不需要安装任何东西，不需要 npm / pnpm / 构建步骤。** 它就是一个静态 HTML 页面。

三种打开方式，任选其一：

### 1. 直接打开（最简单）

双击 `index.html`，或在浏览器里打开它即可。

> 个别浏览器在 `file://` 协议下会限制本地动态加载 `heic2any.min.js`（仅影响 HEIC 转换，普通 jpg/png/webp 不受影响）。若 HEIC 没反应，用下面的「本地服务器」方式。

### 2. 起一个本地服务器（推荐）

任选一个你机器上已有的工具，在项目目录下执行：

```bash
# 方式 A：Python（macOS / 多数 Linux 自带，无需安装）
python3 -m http.server 8080

# 方式 B：Node（若已装 node）
npx serve .          # 或 npx http-server -p 8080
```

然后浏览器打开 `http://localhost:8080`。

### 3. 部署到 HTTPS（推荐长期使用）

纯静态，把文件推上去即可，零配置：

- **GitHub Pages**：仓库 → Settings → Pages → Source 选 `main` 分支根目录，保存后访问 `https://<用户名>.github.io/trace-photo-overlay/`。
- **Vercel / Cloudflare Pages**：连接仓库或拖拽文件夹，无需任何构建命令。

## 操作流程

1. **导入运动轨迹**（可选）：拖入 `.gpx / .kml / .geojson / .tcx / .fit`，自动平滑成曲线，可调颜色 / 线宽。**可一次选多个文件**，按顺序首尾拼接成一条轨迹（适合码表分段记录的多日骑行，文件列表可上移 / 下移 / 删除调整顺序）。
2. **上传 9 张照片**：点格子单张上传，或「批量上传」一次填满，可拖动交换位置。支持 jpg / png / webp / **HEIC** / 实况照片（本地转换）。
3. **文字标注**（可选，Strava 风格）：标题 / 日期 + 可勾选 **距离 / 时长 / 均速 / 配速 / 爬升**，竖排大数字叠加到指定格、附轨迹缩略图。导入轨迹后自动算出并预填，也可手改。骑行勾「距离+时长+均速」，徒步勾「距离+时长+爬升」，跑步勾「距离+时长+配速」。
4. **导出**：9 格填满后下载 `轨迹九宫格.zip`，含 `1.jpg`…`9.jpg`，按 1→9 顺序发朋友圈即拼成九宫格。

`sample-route.gpx` 是一条示例环线，可直接拖进去试。

## 支持的轨迹格式

来自 Strava / 佳明(Garmin) / 行者 / 黑鸟单车 / 两步路 / 顽鹿码表 / 高驰等 App 导出的：

| 格式 | 类型 | 记录的信息 |
|---|---|---|
| **GPX** | 文本 | 一串带时间戳 `<time>` 和海拔 `<ele>` 的经纬度点（`trkpt` / `rtept`），最通用 |
| **KML** | 文本 | 轨迹在 `gx:Track`（带 `when` 时间、海拔）或 `LineString` / `Polygon` 里 |
| **TCX** | 文本 | 佳明格式，`Trackpoint` 含经纬度、海拔、时间 |
| **FIT** | 二进制 | 佳明 / 顽鹿等码表格式，最紧凑、信息最全；码表通常自动暂停，数据最干净 |
| **GeoJSON** | 文本 | `LineString` / `MultiLineString`，常只有坐标（可能无时间 / 海拔） |
| CSV / 纯文本 | 文本 | 每行 `lat,lng` 或 `lng,lat`，自动判别顺序 |

**可一次导入多个文件，按顺序首尾拼接成一条轨迹**——适合码表把一次长途分成多段记录（如多日骑行）。导入后在文件列表里上移 / 下移 / 删除即可调整拼接顺序。直接导入你自己的轨迹即可，**无需改代码**。

## 数据与算法（为什么可能和你的 App 不一样）

**各数字怎么算的：**

- **距离** — 相邻 GPS 点直线距离用 Haversine 公式累加。最准，各文件都能算。
- **时长（运动时长）** — 逐段累加时间，但相邻点速度低于 `0.5 m/s` 的停顿段（等红灯 / 休息 / 过夜）不计入。
- **均速 / 配速** — 距离 ÷ 运动时长。
- **总爬升** — 先对海拔做移动平均平滑（窗口 ±5 点）去掉 GPS 抖动，再累计「相对参考点上升超过 5 米」的部分。直接累加每点正高差会因噪声虚高数倍（某条实测轨迹裸累加 4402m，去噪后 684m，与 App 一致）。

**为什么和 Strava / 两步路 / 码表对不上：**

- **导出有损耗**：导出成 GPX / KML 时，App 内部的精确停顿、气压计海拔等会丢失或简化。
- **算法各家不同**：停顿判定、爬升口径都是各 App 的专有逻辑。例如两步路不自动暂停（停下还在记），码表自动暂停，两者「运动时长」能差几小时；这种情况下本工具去噪后的值反而更贴近你真正在运动的时间。
- **海拔数据源不同**：气压计测的爬升比 GPS 海拔准，而导出文件里通常只有 GPS 海拔。

> **距离总能算准；时长 / 爬升是合理估算。** 满意就用算出来的，不满意就在输入框直接改成你 App 里的官方值——所有数字都能手填。算时长 / 均速 / 配速需要点带时间戳 `<time>`，算爬升需要带海拔 `<ele>`；纯手画的规划路线只有坐标，就只能得到距离。

## 技术实现

单 HTML 文件，关键算法已用 Node 单测验证（`core.mjs` / `fit.mjs` + 对应测试，38 个用例），并用真实多日轨迹（60407 点 GPX、68546 点 FIT）实测校准：

- **轨迹解析**：GPX / KML / TCX / GeoJSON / CSV（文本）+ **FIT（自写二进制解析器，零依赖）**，自动判别经纬度顺序，多文件按序拼接。
- **Catmull-Rom 样条**：稀疏轨迹点加密到 ~500 点，消除折线尖角。
- **Web Mercator 投影**：等比映射到正方形画布，5% padding，保持真实形状不拉伸。
- **九宫格合成**：轨迹渲染成 2400×2400 后切 9 块叠到照片，支持「裁切 1:1」与「保持原比例」。
- **运动数据卡**：Strava 风格竖排（标签 + 大数字）。距离（Haversine）、运动时长（扣停顿）、均速、配速、总爬升（海拔平滑 + 迟滞阈值去噪）从轨迹的 `time` / `ele` 自动算出，指标可勾选，字号随画布自适应。
- **零依赖 ZIP**：自实现 STORE 模式 ZIP 打包（含 CRC32），不引入 JSZip。
- **离线 HEIC**：`heic2any`（libheif WASM 以 data URI 内联）本地托管于 `heic2any.min.js`，无任何外部网络请求。
